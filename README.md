# FRAME · AI-анализ смет на ремонт

> **FRAME видит смету насквозь и защищает хозяина от переплаты.**
> Вставил смету подрядчика — за секунды получил «второе мнение»: где цены честные, а где завышены и на сколько.

ML-модуль цифровой платформы **FRAME** (суперапп для управления ремонтом). Это фича 5.2 продукта — «AI-анализ смет».
Заявка на **Junior ML Contest ITMO** (AI Talent Hub). Автор: Артемий Вербицкий.

🔗 Демо: `python src/demo_app.py` → http://localhost:7860 (или через Docker, см. ниже)

---

## Проблема

Хозяин ремонта получает смету на 200 000 ₽ и не понимает — это честно или его разводят. Рынок непрозрачный, спросить не у кого, а ошибка стоит десятки тысяч. В России нет AI-native решения для этого (на Западе есть: Kukun, Handoff, BidCompareAI).

## Что делает продукт

- смотрит **каждую строку** сметы;
- сравнивает с **реальными рыночными ценами**, собранными у 22 ремонтных компаний по 7 городам;
- помечает завышенные позиции 🚩 и считает **итоговую переплату**;
- флажит **только уверенные** отклонения — не поднимает ложную тревогу.

## Результаты

| Что | Метрика |
|---|---|
| Реальных цен собрано | **2108** (22 компании, 7 городов) |
| Модель справедливой цены (регрессия) | **R² 0.74**, MAPE ~37% |
| Модель цены — domain shift (leave-one-company-out, 22 компании) | R² 0.70±0.09, MAPE 45±11% |
| Выбор модели | линейная > XGBoost по 5-fold CV (обоснованно, принцип простоты) |
| Детекция аномалий — **реальная** проверка (leave-one-company-out) | **ROC-AUC 0.87**, precision 0.74 |
| Детекция аномалий — стресс-тест (синтетика +25..80%) | ROC-AUC 0.57 (честный предел) |

## Методология (полный ML-пайплайн)

1. **Сбор данных** — свой скрапер (`collect_real_prices.py`) собирает реальные прайс-листы (24 сайта → 6391 цена).
2. **Очистка** — каноникализация к 35 работам, границы правдоподобия, IQR-выбросы → 2108 чистых цен.
3. **EDA** — 6 графиков (`reports/figures/`).
4. **Модель цены** — линейная регрессия vs XGBoost, тюнинг по кросс-валидации, честный выбор модели.
5. **2 сценария валидации** — случайный сплит + leave-one-company-out по всем 22 компаниям (domain shift).
6. **Детекция аномалий** — коридоры P10–P90; проверка на реальных межкомпанейских выбросах (LOCO) + синтетический стресс-тест; ROC/PR-кривые.
7. **Интерпретация** — permutation importance + разбор ошибок по работам.
8. **Трекинг экспериментов** — MLflow.
9. **NLP-эксперимент** — семантический матчинг работ на эмбеддингах (`eval_matcher.py`): честный **негативный** результат — regex точнее (см. `reports/matcher_eval.md`).

> **Почему это ML, а не «промт в LLM»:** нейросеть не знает реальных цен на ремонт по 22 компаниям и 7 городам с валидированной точностью — их надо собрать и обучить модель. Каждый компонент имеет функциональную роль и проверен метриками.

## Тесты

```bash
python -m pytest tests/ -v
```

35 юнит-тестов покрывают три критичных для корректности продукта места:
- каноникализация вида работ по regex-правилам (`clean_prices.to_canonical`) — в т.ч. что порядок правил разруливает пересекающиеся ключевые слова, и что нераспознанный текст не пролезает как false positive;
- парсинг строк сметы в демо (`demo_app.parse_line`, `_num`) — числа с пробелами/запятыми, неполные строки, вердикты «завышено/занижено/в норме»;
- построение эмпирических ценовых коридоров (`train_model.empirical_corridor`) — в т.ч. фолбэк с уровня вида работ на уровень категории при малой выборке (n < 5).

## Как запустить

**Демо через Docker (проще всего):**
```bash
docker build -t frame-smeta .
docker run -p 7860:7860 frame-smeta
# открыть http://localhost:7860
```

**Локально (полный пайплайн):**
```bash
python -m venv .venv
.venv\Scripts\activate            # Windows
pip install -r requirements.txt
python src/collect_real_prices.py  # собрать данные
python src/clean_prices.py         # очистить
python src/train_model.py          # обучить + метрики
python src/demo_app.py             # запустить демо
```

## Дорожная карта: от «холодного старта» к маховику данных

Сегодня модель учится на **публичных** прайсах (решение проблемы холодного старта — продукт полезен с первого дня). На платформе каждый проект даёт **собственные** данные, и AI умнеет:

- **Стадия 2:** реальные сметы с объёмами → ловим не только цену, но и **завышенные объёмы и лишние работы**.
- **Стадия 3:** исходы проектов → **прогноз перерасхода/просрочки**, риск-скоринг подрядчиков.
- **Стадия 4:** фотоотчёты, отзывы, споры → **детекция дефектов по фото**, реальная детекция фейк-отзывов, настоящие метки «завышено/честно» из арбитража.

> Данные сметы/исходов/споров рождаются **только внутри платформы** → это ров, который конкурент не скопирует.

## Честные ограничения (= пункты дорожной карты)

- Проверяем цену за единицу, но пока не объёмы (нужны реальные сметы → Стадия 2).
- Метка «аномалия» = отклонение от рынка, а не доказанное мошенничество (арбитраж даст реальные метки → Стадия 4).
- Данные — прайс-листы, не живые сметы (сетевой эффект платформы).

## Как в проекте применялся ИИ (разработка)

Проект собран с интенсивным использованием **AI-агента (Claude Code)** как инструмента разработки: планирование задач, написание скрапера и ML-пайплайна, проведение экспериментов, критический разбор гипотез (в т.ч. отказ от нейроэмбеддингов после честной проверки). Это осознанный современный workflow, а не «весь код руками».
